# SDD 工作流

本项目中的 SDD 指规格驱动开发。它把产品想法转成受控实现，并通过明确测试和评审降低返工。

## 目录模型

```text
sdd/
  SKILL.md
  constitution.md
  workflow.md
  reference/
    prd.md
    architecture.md
  templates/
    spec-template.md
    task-template.md
    test-template.md
    review-template.md
  changes/
    0001-example-change/
      spec.md
      task.md
      test.md
      review.md
```

## Change 创建边界

只有明确进入产品功能开发或系统行为变更时，才创建 `sdd/changes/<change-id>/`。

需要创建 change 的典型场景：

- 用户要求开发新的后端接口、数据库迁移、小程序页面或业务流程。
- 用户要求修改已有功能行为、数据模型、API 契约或 UI 交互。
- 用户要求修复会影响代码行为的缺陷。

不需要创建 change 的典型场景：

- 用户只是询问某个概念、设计理由、表结构考虑、面试问法或命令含义。
- 用户只是排查 Navicat、Docker Desktop、端口占用、账号密码、环境变量等本地环境问题。
- 用户只是要求解释当前功能怎么验收、怎么联调、怎么操作。
- 用户只是要求极小的文档或流程规则修正。

如果一次请求同时包含答疑和代码变更，只为代码变更部分创建 change；答疑内容直接在回复中说明。

## 阶段一：sdd-understand

目标：把请求转换成可以实现的规格。

输入：

- 用户明确要求实现的功能请求。
- `sdd/reference/prd.md`。
- `sdd/reference/architecture.md`。
- 现有代码和测试。

输出：

- `sdd/changes/<change-id>/spec.md`。

必须包含：

- 背景和目标。
- 范围内和范围外。
- 用户故事或用户流程。
- 功能需求。
- 数据模型影响。
- API 影响。
- UI 交互影响。
- 验收标准。
- 风险和开放问题。

退出标准：

- 开发者不需要猜测主要行为即可实现。
- 不确定点已经显式列出。

## 阶段二：sdd-develop

目标：按照 `spec.md` 和 `task.md` 实现。

输入：

- `spec.md`。
- `task.md`。
- 现有代码。

规则：

- 生产行为遵守红绿重构。
- 任务要小，并且可以独立验证。
- 开发过程中同步更新 `task.md`。
- 不得静默扩大范围。

退出标准：

- 所有计划任务已完成或明确延期。
- 代码可以编译。
- 与变更相关的本地测试通过。

## 阶段三：sdd-test

目标：证明功能满足验收标准。

输出：

- 带计划和实际结果的 `test.md`。

测试类别：

- 单元测试。
- API 测试。
- 集成测试。
- UI 或人工流程检查。
- 回归检查。
- 边界条件。

退出标准：

- 每条验收标准都有对应测试或人工检查。
- 失败项已修复，或作为未解决风险记录。

## 阶段四：sdd-review

目标：评估测试通过之外的质量。

输出：

- `review.md`。

评审类别：

- PRD 对齐。
- 架构对齐。
- 模块边界。
- 数据正确性。
- API 契约质量。
- 安全和隐私。
- 错误处理。
- 性能。
- 测试覆盖。
- 可维护性。

退出标准：

- 发现项已修复、明确接受或延期。
- 评审结论已记录。

## 阶段五：sdd-commit-push

目标：准备源码管理交接，同时保留用户控制权。

规则：

- AI 可以执行只读 git 命令。
- AI 可以提出暂存文件范围和提交信息。
- 没有用户明确批准，AI 不得提交或推送。

输出：

- 变更文件摘要。
- 建议提交信息。
- 如用户要求，补充推送或 PR 说明。
