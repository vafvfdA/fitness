# SDD 宪章

本文件定义 AI 和开发者在本项目中必须遵守的规则。

## 不可违反的规则

1. 回答和文档默认使用中文。
   - AI 的用户回复必须使用中文。
   - Markdown 文档中的描述性内容必须使用中文。
   - 代码标识、接口路径、表名、字段名、命令、依赖名、错误码等技术契约可以保留英文。

2. PRD 和架构文档是源参考。
   - 产品意图保存在 `sdd/reference/prd.md`。
   - 架构意图保存在 `sdd/reference/architecture.md`。
   - 具体功能开发应引用这些文件，而不是在每个 change 中完整复制。

3. 只有明确的产品功能开发或行为变更才需要 change 目录。
   - 路径：`sdd/changes/<change-id>/`。
   - 必备文件：`spec.md`、`task.md`、`test.md`、`review.md`。
   - 用户明确要求“开发某个功能”“实现某个接口”“修改某个业务行为”时，必须创建或更新 change 目录。
   - 用户只是询问概念、排查错误、理解命令、咨询连接方式、解释实现思路、讨论面试问题时，不创建 change 目录。
   - Docker、Navicat、Git、环境变量等开发环境答疑或排错，默认记录在对话中即可；只有涉及仓库内配置或代码行为变更时，才按实际变更判断是否需要 change。
   - 极小的文档修正、错别字修正、流程规则修正可以跳过 change 目录，但必须在回复中说明原因。

4. 先理解，再实现。
   - `spec.md` 必须定义范围、用户流程、数据影响、API 影响和验收标准。
   - `task.md` 必须拆成有顺序、可验证的开发步骤。

5. 开发必须遵守红绿重构。
   - 先编写或更新一个失败测试。
   - 确认测试因预期原因失败。
   - 实现让测试通过的最小变更。
   - 只有测试通过后才重构。
   - 任何例外都需要用户明确批准。

6. 测试和评审是两个独立关口。
   - `test.md` 回答：功能有没有做对。
   - `review.md` 回答：设计和实现质量是否足够好。
   - 测试通过不能替代评审。

7. Git 操作必须经过用户审核。
   - AI 可以查看 git 状态并准备提交信息草案。
   - 未经用户明确批准，AI 不得执行 `git commit`、`git push`、分支发布或 PR 创建。
   - 如果用户要求提交，AI 必须先总结变更文件和拟定提交信息。

8. 保护用户已有工作。
   - 不得回滚、删除或覆盖 AI 未创建的改动，除非用户明确要求。
   - 不得在没有明确批准的情况下执行 reset、clean、递归删除等破坏性命令。

9. 控制范围。
   - 不做无关重构。
   - 不在没有说明必要性的情况下引入框架、基础设施或依赖。
   - 优先遵守项目已有约定，而不是新建抽象。

10. 健康和健身安全。
    - 应用可以记录、计算，并基于用户数据给出建议。
    - 应用不得把热量、训练、饮食建议描述成医疗诊断。
    - 知识库回答涉及疾病、伤病、饮食障碍、极端减脂等高风险场景时，必须表达不确定性，并建议咨询专业人士。

11. 隐私和安全。
    - 身体指标、饮食记录、训练记录都属于敏感个人数据。
    - 不得无必要记录 token、密码、OpenID、手机号或私密健康信息。
    - API 必须强制用户级数据隔离。

## 完成定义

一个变更只有满足以下条件才算完成：

- `spec.md` 和用户请求一致。
- `task.md` 中所有实现任务已完成，或明确标记为延期。
- `test.md` 记录了测试命令、人工检查、结果和未解决风险。
- `review.md` 记录了评审发现和最终结论。
- 最终回复说明改了什么、测了什么、还有什么风险。
