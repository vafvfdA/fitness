# 0001 训练饮食 MVP 规格

## 背景

本变更定义第一个可开发的 MVP，依据如下：

- `sdd/reference/prd.md`
- `sdd/reference/architecture.md`

MVP 应先建立训练和饮食记录闭环，再进入高级 AI 知识库能力。

## 目标

用户可以记录每日训练和饮食，查看紧凑日历摘要，并维护基础体重目标信息。

## 范围内

- 基础用户身份，可以先使用开发模式用户，或预留微信登录边界。
- 创建和查看每日训练记录。
- 使用训练模板快速录入。
- 创建和查看每日饮食记录。
- 使用食物模板快速录入。
- 月度日历摘要。
- 基础个人资料：当前体重、目标体重、距离目标差值。
- 基础训练计划：训练天数、休息天数、部位轮换。
- 后端表结构、CRUD API、参数校验和测试。
- 微信小程序核心页面。

## 范围外

- 向量知识库。
- AI 流式问答。
- 穿戴设备接入。
- 社交分享。
- 支付或订阅。
- 复杂营养宏量计算。
- 医疗或伤病诊断。

## 用户流程

### 训练记录

1. 用户打开今日页。
2. 用户进入选定日期的训练页面 A。
3. 用户选择训练模板，或创建自定义动作。
4. 用户编辑动作组数、次数、重量、时长和预估热量。
5. 系统更新总组数和预估消耗热量。
6. 用户保存训练记录。
7. 用户可以跳到同一天的饮食页面 B。

### 饮食记录

1. 用户打开选定日期的饮食页面 B。
2. 用户选择餐次。
3. 用户选择食物模板，或输入自定义食物。
4. 用户输入数量和热量。
5. 系统更新摄入总热量。
6. 用户保存饮食记录。

### 日历摘要

1. 用户打开日历页面。
2. 系统展示月份格子。
3. 有记录的日期展示训练部位短标签和预估消耗热量。
4. 用户点击日期进入当天详情。

### 计划和体重目标

1. 用户打开计划页面。
2. 用户设置当前体重和目标体重。
3. 系统展示距离目标还差多少。
4. 用户配置训练和休息周期、部位轮换。
5. 系统展示今日计划训练标签。

## 功能需求

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| FR-1 | 用户可以按日期创建、更新、查询每日训练记录。 | 必须 |
| FR-2 | 用户可以在训练记录下添加多个动作。 | 必须 |
| FR-3 | 用户可以在每个动作下添加多组记录。 | 必须 |
| FR-4 | 系统根据组记录计算训练总组数。 | 必须 |
| FR-5 | 系统保存并展示预估训练消耗热量。 | 必须 |
| FR-6 | 用户可以创建训练模板，并把模板应用到某一天。 | 必须 |
| FR-7 | 用户可以按日期创建、更新、查询每日饮食记录。 | 必须 |
| FR-8 | 用户可以按餐次添加食物记录。 | 必须 |
| FR-9 | 系统根据食物条目计算饮食摄入总热量。 | 必须 |
| FR-10 | 用户可以创建食物模板，并按数量应用。 | 应该 |
| FR-11 | 日历 API 返回紧凑的月度摘要。 | 必须 |
| FR-12 | 用户可以设置当前体重和目标体重。 | 必须 |
| FR-13 | 系统计算距离目标体重的差值。 | 必须 |
| FR-14 | 用户可以配置训练休息周期和部位轮换。 | 应该 |
| FR-15 | API 必须按认证用户隔离数据。 | 必须 |

## 数据影响

使用 `sdd/reference/architecture.md` 中建议的数据表：

- `user_account`
- `user_profile`
- `body_metric_record`
- `workout_record`
- `workout_exercise_entry`
- `workout_set_entry`
- `workout_template`
- `workout_template_item`
- `diet_record`
- `diet_food_entry`
- `food_template`
- `training_plan`

知识库相关表可以后续变更再创建，除非需要提前存放种子数据。

## API 影响

初始接口：

- `POST /api/v1/auth/wechat-login`
- `GET /api/v1/profile`
- `PUT /api/v1/profile`
- `POST /api/v1/workouts`
- `GET /api/v1/workouts/by-date?date=YYYY-MM-DD`
- `PUT /api/v1/workouts/{id}`
- `DELETE /api/v1/workouts/{id}`
- `POST /api/v1/workout-templates/{id}/apply`
- `POST /api/v1/diets`
- `GET /api/v1/diets/by-date?date=YYYY-MM-DD`
- `PUT /api/v1/diets/{id}`
- `DELETE /api/v1/diets/{id}`
- `POST /api/v1/food-templates`
- `GET /api/v1/calendar/monthly-summary?month=YYYY-MM`
- `GET /api/v1/plans/current`
- `PUT /api/v1/plans/current`

## UI 影响

必需页面：

- 今日页，包含训练和饮食快捷卡片。
- 训练页面 A。
- 饮食页面 B。
- 日历页面。
- 计划页面。
- 个人资料或基础设置页面。

交互要求：

- 训练和饮食页面之间跳转时，选中日期必须保持一致。
- 保存后，当日摘要数字必须更新。
- API 提交前必须有可见的表单校验。
- 空状态要引导用户添加模板或记录。

## 验收标准

| 编号 | 标准 |
| --- | --- |
| AC-1 | 用户可以保存一次包含至少 2 个动作、每个动作多组记录的训练。 |
| AC-2 | 训练详情展示动作数量、总组数和预估消耗热量。 |
| AC-3 | 用户可以保存一条包含至少 2 个餐次食物的饮食记录。 |
| AC-4 | 饮食详情展示摄入总热量。 |
| AC-5 | 日历月份视图能在有记录的日期展示部位短标签和训练热量。 |
| AC-6 | 计划页面展示当前体重、目标体重和差值。 |
| AC-7 | 一个用户不能查询另一个用户的数据。 |
| AC-8 | 后端校验会拒绝非法日期、负数热量和非法组数。 |

## 风险

- 微信小程序流式支持不属于本 MVP，但会影响后续知识库设计。
- 热量估算必须在产品文案上表达为近似值。
- 如果日历格子展示过多信息，移动端会拥挤。
- 训练模板在真实使用后可能需要调整数据模型。

## 开放问题

- MVP 是否立即接入真实微信登录，还是先用开发模式用户身份？
- 食物热量第一版是纯手动录入，还是预置系统食物模板？
- 体重在 MVP 中是否只保存在 profile，还是同时记录历史？
