# 0006 训练记录后端接口规格

## 背景

PRD 中的 A 界面需要记录每天练了什么、几个动作、每个动作几组，以及估算消耗热量；日历页面需要按天展示训练部位和消耗热量。当前项目已经有训练相关数据库表，但还没有后端业务接口，前后端无法围绕训练记录做联调。

## 目标

提供训练记录的最小后端闭环：新增一次训练记录、查询某天训练记录、查询某月日历摘要。接口返回结构要稳定，便于后续微信小程序 A 界面和日历页面直接接入。

## 范围内

- 新增训练记录 API。
- 查询指定日期训练记录 API。
- 查询指定月份训练日历摘要 API。
- 训练记录、动作、组明细的 JDBC 持久化。
- 请求参数校验和基础错误响应。
- 自动化测试覆盖核心接口行为。

## 范围外

- 微信登录和正式鉴权。
- 饮食记录接口。
- 训练模板接口。
- 小程序页面联调。
- 真实热量算法，只记录前端或用户输入的估算值。

## 用户流程

1. 小程序在 A 界面录入训练日期、训练部位、动作、每个动作的组明细和估算热量。
2. 小程序调用新增训练记录接口，后端保存一次训练及其动作、组明细。
3. 小程序进入当天训练详情时，调用按日期查询接口展示当天训练。
4. 小程序进入日历页时，调用月历摘要接口展示每天的训练部位、总组数和估算消耗热量。

## 功能需求

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| FR-1 | 支持新增训练记录，包含训练主信息、动作列表和组明细 | 必须 |
| FR-2 | 后端根据组明细数量计算 `totalSets`，不信任前端传入总组数 | 必须 |
| FR-3 | 支持按 `date` 查询某个用户当天的训练记录列表 | 必须 |
| FR-4 | 支持按 `month` 查询某个用户的训练日历摘要 | 必须 |
| FR-5 | 接口使用统一响应结构 `ApiResponse` | 必须 |
| FR-6 | 登录体系完成前，接口临时使用 `X-User-Id` 请求头识别用户 | 必须 |

## 数据影响

复用既有表：

- `workout_record`：保存一次训练主记录。
- `workout_exercise_entry`：保存训练中的动作。
- `workout_set_entry`：保存每个动作的组明细。

本变更不新增迁移脚本。记录创建时按父子层级插入，查询时按 `workout_record -> workout_exercise_entry -> workout_set_entry` 聚合返回。

## API 影响

### 新增训练记录

`POST /api/v1/workout-records`

请求头：

- `X-User-Id`: 当前用户 ID，临时联调字段。

请求体：

```json
{
  "workoutDate": "2026-07-20",
  "title": "胸肩训练",
  "bodyPart": "胸",
  "estimatedCalories": 320,
  "note": "状态不错",
  "exercises": [
    {
      "exerciseName": "卧推",
      "bodyPart": "胸",
      "estimatedCalories": 180,
      "note": "最后一组力竭",
      "sets": [
        {
          "setNo": 1,
          "reps": 10,
          "weightKg": 60.0
        }
      ]
    }
  ]
}
```

响应体 `data` 返回保存后的完整训练记录，包含数据库 ID 和后端计算的 `totalSets`。

### 查询某天训练记录

`GET /api/v1/workout-records?date=2026-07-20`

请求头：

- `X-User-Id`: 当前用户 ID。

响应体 `data` 为训练记录列表。

### 查询月历摘要

`GET /api/v1/workout-records/calendar?month=2026-07`

请求头：

- `X-User-Id`: 当前用户 ID。

响应体 `data`：

```json
{
  "month": "2026-07",
  "days": [
    {
      "date": "2026-07-20",
      "bodyParts": ["胸", "肩"],
      "totalSets": 8,
      "estimatedCalories": 320,
      "workoutCount": 1
    }
  ]
}
```

## UI 影响

本变更不直接修改小程序页面，但返回结构需要满足后续页面：

- A 训练详情页可以直接渲染动作和组明细。
- 日历页可以直接展示日期格子中的训练部位和估算消耗热量。

## 验收标准

| 编号 | 标准 |
| --- | --- |
| AC-1 | 新增训练记录后，响应包含记录 ID、动作 ID、组 ID 和正确的总组数 |
| AC-2 | 按日期查询只返回当前用户和指定日期的训练记录 |
| AC-3 | 月历摘要能按天聚合训练部位、总组数、估算热量和训练次数 |
| AC-4 | 缺少必填字段时返回 400 和统一错误响应 |
| AC-5 | `mvn test` 通过 |

## 风险

- 当前还没有登录体系，`X-User-Id` 只是开发期联调方案，后续需要替换为真实认证上下文。
- 当前热量为估算录入值，不代表医学或运动科学上的精确消耗。

## 开放问题

- 训练部位后续是否固定枚举，需要结合小程序输入控件再决定。
