# Feature Specification: 饮食记录后端接口

**Feature Branch**: `001-diet-record-api`

**Created**: 2026-07-20

**Status**: Draft

**Input**: 用户要求在完成项目宪章后，开始下一个功能开发。结合既有 PRD，下一个功能为 B 界面的饮食记录后端接口。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 录入当天饮食记录 (Priority: P1)

用户在 B 界面录入某天吃了什么、每种食物的数量、单位、热量和营养素信息，系统保存为当天饮食记录，并返回保存后的完整明细。

**Why this priority**: 饮食记录是 B 界面的核心输入能力，没有新增记录就无法做热量管理和后续小程序联调。

**Independent Test**: 调用新增饮食记录接口，提交日期、备注和食物明细，响应包含记录 ID、食物明细 ID、用户 ID、日期和后端计算的总热量。

**Acceptance Scenarios**:

1. **Given** 用户填写了早餐和午餐食物，**When** 用户保存饮食记录，**Then** 系统保存记录并返回总热量和食物明细。
2. **Given** 用户填写的单项食物热量为 350 和 420，**When** 用户保存饮食记录，**Then** 系统返回当天记录总热量为 770。

---

### User Story 2 - 查询某天饮食记录 (Priority: P2)

用户进入某天的 B 界面时，可以看到该日期已经录入的饮食记录和食物明细。

**Why this priority**: 查询能力是编辑、回看和小程序页面回显的基础。

**Independent Test**: 分别为不同用户和不同日期创建饮食记录，按日期查询时只返回当前用户指定日期的数据。

**Acceptance Scenarios**:

1. **Given** 当前用户在 2026-07-20 和 2026-07-21 都有记录，**When** 查询 2026-07-20，**Then** 只返回 2026-07-20 的记录。
2. **Given** 其他用户在同一天有记录，**When** 当前用户查询当天记录，**Then** 不返回其他用户的数据。

---

### User Story 3 - 查看某天饮食汇总 (Priority: P3)

用户需要快速知道某天吃了多少热量，以及蛋白质、脂肪、碳水的大概摄入量。

**Why this priority**: 日汇总能支撑后续计划页、日历页和热量目标对比。

**Independent Test**: 为某天创建多条饮食记录和多个食物明细，查询汇总时返回总热量、蛋白质、脂肪、碳水和食物数量。

**Acceptance Scenarios**:

1. **Given** 用户当天录入多种食物，**When** 查询饮食汇总，**Then** 系统返回当天总热量和三大营养素合计。
2. **Given** 用户当天没有饮食记录，**When** 查询饮食汇总，**Then** 系统返回 0 值汇总而不是错误。

### Edge Cases

- 缺少饮食日期、餐次、食物名称、数量、单位或食物列表时，系统返回 400 和统一错误响应。
- 食物热量、数量和营养素不能为负数。
- 查询没有数据的日期时，列表为空，汇总为 0。
- 不同用户的数据必须隔离。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须支持新增饮食记录，包含饮食日期、备注和食物明细列表。
- **FR-002**: 系统必须根据食物明细的热量计算饮食记录 `totalCalories`，不信任前端传入总热量。
- **FR-003**: 系统必须保存每条食物明细的餐次、食物名称、数量、单位、热量、蛋白质、脂肪和碳水。
- **FR-004**: 用户必须能够按日期查询自己的饮食记录列表。
- **FR-005**: 用户必须能够按日期查询自己的饮食汇总。
- **FR-006**: 系统必须按用户隔离饮食记录，不能返回其他用户的数据。
- **FR-007**: 系统必须使用统一响应结构返回成功和校验错误。
- **FR-008**: 登录体系完成前，系统允许通过开发期请求头识别用户。

### Key Entities *(include if feature involves data)*

- **饮食记录**: 表示某个用户某天的一次饮食记录，包含日期、总热量、备注和食物明细。
- **食物明细**: 表示饮食记录中的一个食物条目，包含餐次、食物名称、数量、单位、热量和营养素。
- **饮食汇总**: 表示某个用户某天的合计摄入结果，包含总热量、蛋白质、脂肪、碳水和食物数量。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在一次请求中保存至少 10 条食物明细，并得到完整保存结果。
- **SC-002**: 查询某天饮食记录时，响应只包含当前用户和指定日期的数据。
- **SC-003**: 查询某天饮食汇总时，即使当天没有记录，也能返回 0 值汇总。
- **SC-004**: 自动化测试覆盖新增、按日期查询、日汇总和参数校验。

## Assumptions

- 当前阶段不实现微信登录，继续临时使用 `X-User-Id` 请求头传递用户 ID。
- 当前阶段不接入食物热量知识库，食物热量由用户或前端传入。
- 当前阶段不做更新和删除接口，只完成新增、查询和汇总的最小闭环。
