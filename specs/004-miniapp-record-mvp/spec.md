# Feature Specification: 小程序记录 MVP 联调

**Feature Branch**: `004-miniapp-record-mvp`

**Created**: 2026-07-21

**Status**: Draft

**Input**: 用户确认下一步开发微信小程序前端 MVP，优先打通今日汇总、训练模板生成训练记录、饮食录入和页面互跳。

## User Scenarios & Testing

### User Story 1 - 查看今日汇总 (Priority: P1)

用户打开小程序首页后，可以看到指定日期的训练数量、训练组数、训练消耗、饮食摄入和净热量。

**Why this priority**: 首页是判断训练和饮食是否记录成功的入口，也是后续 A/B 页面联调结果的反馈面板。

**Independent Test**: 后端存在当天训练和饮食数据时，首页显示汇总数据；无数据时显示 0 值。

**Acceptance Scenarios**:

1. **Given** 用户打开今日页，**When** 页面加载，**Then** 系统请求今日汇总并展示训练和饮食指标。
2. **Given** 用户从训练页或饮食页返回今日页，**When** 页面重新显示，**Then** 今日汇总刷新。

---

### User Story 2 - 使用训练模板生成训练记录 (Priority: P2)

用户进入训练页后，可以查看当天训练记录和自己的训练模板；如果没有模板，可以创建一个示例模板；选择模板后生成当天训练记录。

**Why this priority**: 这是 A 界面的最小可用闭环，能验证模板接口和训练记录接口是否适合手机端录入。

**Independent Test**: 用户在训练页创建或选择模板后，当天训练记录列表新增记录，并能跳转到饮食页。

**Acceptance Scenarios**:

1. **Given** 用户没有训练模板，**When** 点击创建示例模板，**Then** 页面出现可选模板。
2. **Given** 用户选择模板，**When** 生成训练记录成功，**Then** 页面刷新当天训练记录列表。
3. **Given** 用户点击查看饮食，**When** 页面跳转，**Then** 饮食页保留同一天日期。

---

### User Story 3 - 录入饮食记录 (Priority: P3)

用户进入饮食页后，可以查看当天饮食记录，并通过表单录入食物名称、餐次、数量、单位、热量和三大营养素。

**Why this priority**: 这是 B 界面的最小可用闭环，能验证饮食记录接口和今日汇总接口。

**Independent Test**: 用户提交食物后，当天饮食记录列表新增记录，摄入热量汇总变化。

**Acceptance Scenarios**:

1. **Given** 用户填写食物信息，**When** 提交表单，**Then** 系统创建当天饮食记录并刷新列表。
2. **Given** 用户未填写食物名称，**When** 提交表单，**Then** 页面提示补全信息且不发起无效提交。
3. **Given** 用户点击查看训练，**When** 页面跳转，**Then** 训练页保留同一天日期。

### Edge Cases

- 后端服务未启动或请求失败时，页面必须显示失败提示，不应白屏。
- 接口返回空数组时，页面显示空状态。
- 输入数字为空时，前端按 0 或合理默认值处理，避免提交非法数据。
- 开发期用户固定为 `1001`，后续登录完成后替换。
- 当前日期默认使用本地日期，支持从页面参数传入指定日期。

## Requirements

### Functional Requirements

- **FR-001**: 今日页必须请求并展示今日汇总。
- **FR-002**: 请求工具必须统一携带 `X-User-Id`。
- **FR-003**: 训练页必须查询当天训练记录。
- **FR-004**: 训练页必须查询训练模板列表。
- **FR-005**: 训练页必须支持创建一个开发期示例训练模板。
- **FR-006**: 训练页必须支持从模板生成当天训练记录。
- **FR-007**: 饮食页必须查询当天饮食记录。
- **FR-008**: 饮食页必须支持新增饮食记录。
- **FR-009**: 训练页和饮食页必须支持携带日期互跳。
- **FR-010**: 所有页面必须对加载、空数据和请求失败提供可见状态。

### Key Entities

- **今日汇总视图**: 展示训练、饮食和净热量指标。
- **训练记录视图**: 展示当天训练记录、动作数量、组数和消耗。
- **训练模板视图**: 展示可用于生成训练记录的模板。
- **饮食记录视图**: 展示当天饮食记录和食物明细。
- **饮食表单**: 收集食物、餐次、数量、单位、热量和营养素。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户可以在微信开发者工具中完成一次“创建模板或选择模板 -> 生成训练记录 -> 今日汇总刷新”的流程。
- **SC-002**: 用户可以在微信开发者工具中完成一次“新增饮食 -> 饮食列表刷新 -> 今日汇总刷新”的流程。
- **SC-003**: 今日页、训练页、饮食页都能在无数据时正常展示。
- **SC-004**: 前端请求默认携带开发期用户 ID，后端能完成用户隔离联调。
- **SC-005**: 页面数据映射函数有自动化测试覆盖。

## Assumptions

- 后端服务本地运行在 `http://localhost:8080/api/v1`。
- 微信开发者工具联调时会关闭合法域名校验，或后续再配置本地域名代理。
- 本轮不做微信登录、不做正式 UI 设计系统、不做复杂表单校验。
- 本轮不引入 npm 依赖和跨端框架，继续使用原生微信小程序。
